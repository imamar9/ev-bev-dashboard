# Git Workflow Standards

Rules for all code changes to `ev-dashboard/`. Critical context: **every push to `main` triggers an immediate Netlify production redeploy.** A bad push breaks the live app for everyone with the link.

---

## Branch Strategy

### Never push directly to `main`

`main` is the production branch. All work happens on a feature branch:

```bash
git checkout -b feature/your-feature-name
# do work
git push origin feature/your-feature-name
```

**HARD BLOCK — direct push to `main` is only permitted for:**
- Emergency hotfixes where the live app is broken
- Documentation-only changes (`.md` files, comments)
- In both cases, run the full pre-push checklist first

### Branch naming convention

```
feature/short-description       # new feature
fix/short-description           # bug fix
data/update-pipeline-output     # data refresh (JSON files updated)
chore/short-description         # dependency update, refactor, config
docs/short-description          # documentation only
```

Examples:
```
feature/charging-infrastructure-page
fix/geolocation-https-detection
data/june-2026-ev-population-refresh
chore/update-plotly-3-6
```

## Commit Messages

### Format

```
<type>: <short summary in present tense, max 72 chars>

<optional body — what changed and why, not how>
```

### Types

| Type | Use for |
|------|---------|
| `feat` | New page, new chart, new API route, new chatbot topic |
| `fix` | Bug fix |
| `data` | Pipeline re-run, JSON files updated |
| `chore` | Dependency update, config change, refactor with no behaviour change |
| `docs` | Knowledgebase, README, comments only |
| `style` | CSS/Tailwind changes only, no logic change |

### Examples

```
feat: add charging infrastructure page with utility breakdown chart

fix: show clear message when geolocation blocked on HTTP

data: refresh EV population data to June 2026 export

chore: upgrade plotly.js from 3.5 to 3.6

docs: add data pipeline schema to knowledgebase
```

### Rules
- Subject line: present tense, no period at end, max 72 characters
- Do not write "changed X" or "updated Y" — write "add X" or "update Y"
- Do not reference ticket numbers unless a project management tool is in use
- Do not write "WIP" commits — squash before merging

## Pre-Push Checklist

Run every item before `git push`:

```bash
# 1. TypeScript — catch type errors before Netlify does
npx tsc --noEmit

# 2. Full production build
npm run build

# 3. Data validation (only if JSON files were changed)
python testscripts/validate_outputs.py

# 4. Secrets check
git diff --staged | grep -iE "(api_key|secret|password|token)"

# 5. Check what you're about to push
git diff main...HEAD --stat
```

**HARD BLOCK — do not push if `npm run build` exits non-zero.**

## What to Push vs. What Not to Push

### Never commit these files

| File/pattern | Reason |
|---|---|
| `.env.local` | Contains API keys — already gitignored, but verify |
| `node_modules/` | Reconstructed by `npm install` |
| `.next/` | Build output — reconstructed by `npm run build` |
| `output/api_data/*.json` | Generated by pipeline — source of truth is the CSV + script |
| `inputs/*.csv` | Large source data files — share via other means |

### Always commit these files when changed

| File | Commit when |
|------|------------|
| `ev-dashboard/public/data/api_data/*.json` | After pipeline re-run and validation |
| `ev-dashboard/src/**/*.ts(x)` | After any code change |
| `knowledgebase/*.md` | After updating documentation |
| `.claude/rules/*.md` | After updating development standards |
| `netlify.toml` | After intentional build config changes |
| `testscripts/validate_outputs.py` | After adding new validation rules |

## Merging to Main

### Before merging a feature branch

1. Rebase onto latest main to avoid merge conflicts in production:
   ```bash
   git fetch origin
   git rebase origin/main
   ```
2. Run the full pre-push checklist above
3. Confirm the Netlify preview deploy (if available) works correctly
4. Merge with a descriptive merge commit or fast-forward

### After merging

Watch the Netlify deploy log for 2–3 minutes:
- Go to `app.netlify.com` → your site → Deploys
- Confirm the deploy status shows **Published** (green)
- Open the live URL and confirm the page loads and charts render

## Rollback Procedure

If a bad push breaks production:

### Option 1 — Revert the commit (preferred)

```bash
# Find the bad commit hash
git log --oneline -5

# Revert it (creates a new commit that undoes the bad one)
git revert <commit-hash>
git push origin main
```

Netlify redeploys automatically with the reverted code.

### Option 2 — Netlify instant rollback (fastest)

1. Go to `app.netlify.com` → your site → Deploys
2. Find the last known good deploy
3. Click **"Publish deploy"** on that entry
4. Production rolls back instantly — no git changes needed

### Option 3 — Force push to last known good (last resort)

```bash
git reset --hard <last-good-commit-hash>
git push --force-with-lease origin main
```

Only use this if Option 1 and 2 are not possible. `--force-with-lease` is safer than `--force` — it refuses if someone else has pushed in the meantime.

## Data Refresh Workflow

When updating dashboard data (new CSV, updated pipeline):

```bash
# 1. Run pipeline
python scripts/export_ev_population_data.py

# 2. Validate
python testscripts/validate_outputs.py

# 3. Copy to dashboard
cp output/api_data/*.json ev-dashboard/public/data/api_data/

# 4. Test build
cd ev-dashboard && npm run build

# 5. Commit as a data update (not a feature)
git add ev-dashboard/public/data/api_data/
git commit -m "data: refresh EV population data to <month year>"
git push origin main
```

## Checklist Summary

| When | Check |
|------|-------|
| Starting work | Create a feature branch — never work on `main` |
| Before every commit | `git diff --staged` — no secrets, no build artifacts |
| Before every push | `npm run build` exits 0, `tsc --noEmit` clean |
| After pushing to main | Watch Netlify deploy log until Published |
| If something breaks | Netlify rollback first (fastest), revert commit second |
